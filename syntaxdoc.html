Tutorial

Chapter 1 — Introduction

Welcome to YBM-16 assembly. This tutorial will take you from zero knowledge to writing real programs: graphics, loops, functions, and everything in between. Read every chapter in order. By the end, nothing in this language will be a mystery.

What is assembly?

Most programming languages like Python or C sit far above the hardware. When you write x = 5 + 3, a compiler or interpreter translates that into dozens of tiny machine operations. Assembly is one level above the raw binary — every line you write corresponds to exactly one instruction the CPU executes. There is no magic translation layer. You are in direct control.

This directness is both assembly's power and its challenge. You can do anything, but you have to be explicit about everything.

How YBM-16 works

YBM-16 is a 16-bit CPU. "16-bit" means it works with numbers up to 65,535 (0xFFFF) natively. It has:

  • 17 registers — small, ultra-fast storage slots inside the CPU itself
  • 64KB of RAM — the main working memory, addressed from 0x0000 to 0xFFFF
  • 16KB of VRAM — memory mapped to a 128×128 pixel display
  • 35 instructions — the complete vocabulary of things the CPU can do

Every instruction is exactly 4 bytes wide. The first byte is always the opcode (which instruction), and the remaining 3 bytes carry operands like register numbers or addresses.

Your first program

Let's start with the simplest possible program:

ldi  r0, 42     ; put the number 42 into register r0
halt            ; stop

This loads the number 42 into register r0, then stops. When the CPU halts you can inspect r0 in the debugger and confirm it holds 42. That's the whole program. From here, everything builds on this foundation.

Note Lines starting with ; or anything after a ; on a line are comments. The assembler ignores them completely. Use comments freely — your future self will thank you.

Chapter 2 — Registers

Registers are the CPU's working hands. Before you can do anything else you need to understand them.

The register file

YBM-16 has 17 registers:

NameIDPurpose
r0 – r140x00 – 0x0EGeneral purpose. Use for any data, counters, temporaries.
idx0x0FIndex register. Conventionally used as a pointer or loop counter, but works identically to r0–r14.
sp0x10Stack pointer. Automatically managed by push/pop/call/ret. Avoid writing to it manually unless you know what you're doing.

Registers are not memory

Registers live inside the CPU chip itself. Reading or writing a register takes zero clock cycles of memory access — it's instantaneous. RAM access is much slower. Whenever possible, keep your working data in registers and only go to RAM when you must.

Registers hold 16-bit values

Every register stores exactly one 16-bit unsigned integer: a number from 0 to 65535. There are no "types" — a register doesn't know if it's holding a pixel color, a memory address, or a plain number. That distinction lives entirely in how your program uses it.

r0 convention

r0 is not special to the hardware, but by convention many programs use it as a permanent zero register. If you load zero into r0 at startup and never write to it again, you can use cmpr rx, r0 to test whether rx is zero, which is a very common pattern. 

ldi r0, 0   ; establish zero register
; ... now r0 = 0 throughout the program

Chapter 3 — The Memory Model

Understanding where data lives is essential before you can write programs that do anything interesting.

The address space

YBM-16 has a flat 16-bit address space: addresses 0x0000 through 0xFFFF (65,536 locations, each holding one 16-bit word). Think of it as a giant numbered array of slots: 

Address:  0x0000  0x0001  0x0002  ...  0xFFFF
Content:  [    ] [    ] [    ] ...  [    ]

Layout

RangeRegionDescription
0x0000 – 0x3FFFProgram ROMYour assembled code is loaded here. Usually read-only at runtime.
0x4000 – 0xBFFFRAMFree to use for variables, arrays, stack frames — anything you need.
0xC000 – 0xFFFFVRAMWrites here appear on the 128×128 display. Each address is one pixel.

Variables in RAM

YBM-16 has no variable declarations. If you want a variable, pick an address in RAM and use it consistently. It's conventional to put your variables right after the program code, or at a known high address: 

; Program code at 0x0000...
ldi  r1, 100
st   r1, 0x4000    ; 0x4000 is our "score" variable

; Later...
ld   r2, 0x4000    ; read "score" back

Labels as addresses

Labels in the assembler resolve to byte addresses. You can use a label anywhere an address is expected, which makes your code readable and relocatable: 

; This label points to address 0x0010 (if it's the 5th instruction, 5*4=20=0x14)
loop_start:
    inc  r1
    jmp  loop_start

Chapter 4 — Moving Data

Almost every program starts the same way: loading values into registers to work with. There are four ways to get data into a register.

Method 1: Load an immediate (ldi)

ldi puts a literal number directly into a register. "Immediate" means the value is baked into the instruction itself — it doesn't come from anywhere in memory. 

ldi r1, 255       ; r1 = 255 (decimal)
ldi r2, 0xFF      ; r2 = 255 (hex — same thing)
ldi r3, 0b1010   ; r3 = 10 (binary)

Use ldi for: constants, initial values, setting up counters.

Method 2: Load from memory (ld)

ld reads a 16-bit value from a fixed address in RAM into a register. The address is hardcoded into the instruction. 

ld r1, 0x4000    ; r1 = RAM[0x4000]

Use ld when: you know at compile time which address to read.

Method 3: Load from a pointer (ldind)

ldind (load indirect) treats the value in a register as a memory address, then loads from there. This is pointer dereferencing — essential for arrays and dynamic data. 

ldi   idx, 0x4000  ; idx holds an address
ldind r1, idx     ; r1 = RAM[idx] = RAM[0x4000]
inc   idx          ; advance pointer
ldind r2, idx     ; r2 = RAM[0x4001]

Use ldind when: iterating over arrays, following pointers computed at runtime.

Method 4: Copy from another register (mov)

mov copies one register's value into another. The source is unchanged. 

ldi r1, 42
mov r2, r1    ; r2 = 42, r1 still = 42

Storing back to memory

st and stind are the mirror operations. After computing a result in a register, store it to RAM to persist it: 

ldi  r1, 999
st   r1, 0x4000   ; RAM[0x4000] = 999

ldi   idx, 0x4010
stind r1, idx     ; RAM[idx] = r1
Common mistake Writing ld r1, r2 won't work — ld takes a literal address, not a register. If your address is in a register, you need ldind.

Chapter 5 — Arithmetic

YBM-16 supports the five fundamental arithmetic operations. All of them work exclusively on registers — you cannot do math directly on memory values.

The three-register pattern

All binary arithmetic follows the same form: op dst, s1, s2. The result goes into dst, and the two source registers are unchanged (unless dst is the same as one of them). 

ldi r1, 10
ldi r2, 3
add r3, r1, r2   ; r3 = 13
sub r4, r1, r2   ; r4 = 7
mul r5, r1, r2   ; r5 = 30
div r6, r1, r2   ; r6 = 3  (integer division)
mod r7, r1, r2   ; r7 = 1  (remainder)

In-place operations

You can use the destination as a source too, which is very common for accumulators: 

ldi r1, 5
add r1, r1, r2   ; r1 = r1 + r2 (r1 updates in place)

inc and dec

Incrementing and decrementing by 1 is so common it gets its own instructions. They're shorter to write and slightly faster than add r1, r1, one_reg: 

ldi r1, 0
inc r1    ; r1 = 1
inc r1    ; r1 = 2
dec r1    ; r1 = 1

Negation

neg flips the sign using two's complement. Since YBM-16 is 16-bit, values wrap: neg 1 = 65535 = 0xFFFF, which in signed arithmetic is -1. 

ldi r1, 5
neg r1    ; r1 = 0xFFFB (-5 in signed 16-bit)
Tip — computing with constants The CPU has no "add immediate" instruction. To add a constant, load it into a spare register first: 
ldi r14, 10
add r1, r1, r14
Many programs dedicate a register (like r14) to frequently used small constants.

Chapter 6 — Bitwise Logic

Bitwise operations treat a 16-bit value as 16 individual bits and act on each independently. They're essential for flags, masks, packing/unpacking color channels, and many other low-level tasks.

AND — masking

AND sets a bit to 1 only if both corresponding bits are 1. Its main use is extracting specific bits by applying a "mask": 

; Extract the lower 4 bits (nibble) of r1
ldi  r2, 0x000F
gand r3, r1, r2   ; r3 = lower nibble only

OR — setting bits

OR sets a bit if either input has it set. Use it to force specific bits on:

; Set bit 3 of r1 without touching anything else
ldi r2, 0x0008
gor r1, r1, r2

XOR — toggling / clearing

XOR flips bits. XORing a value with itself always produces zero — a fast way to zero a register:

gxor r1, r1, r1   ; r1 = 0, regardless of previous value

; Toggle a flag in r2
ldi  r3, 0x0001
gxor r2, r2, r3   ; flip bit 0

NOT — invert all bits

ldi  r1, 0xFF00
gnot r1           ; r1 = 0x00FF

Shifts — multiply/divide by powers of 2

Shifting left by N is the same as multiplying by 2^N. Shifting right divides. This is faster than mul or div for powers of two: 

ldi r1, 1
ldi r2, 4
shl r3, r1, r2   ; r3 = 1 << 4 = 16
shr r4, r3, r2   ; r4 = 16 >> 4 = 1

Practical: Packing two bytes into one word

Since YBM-16 works with 16-bit words, you often pack two 8-bit values together:

; Pack r1 (high byte) and r2 (low byte) into r3
ldi  r4, 8
shl  r3, r1, r4   ; shift r1 up 8 bits
gor  r3, r3, r2   ; combine with r2

; Unpack: extract low byte
ldi  r5, 0x00FF
gand r6, r3, r5   ; r6 = low byte

Chapter 7 — Control Flow

Without branching, every program just falls through from top to bottom once. Jumps and comparisons let you make decisions, creating the logic that makes programs useful.

How flags work

The CPU has three internal status flags. You cannot read them directly into a register, but they are set automatically by cmpr and tested by conditional jumps:

FlagWhen it's set
Z (zero)The result of the last cmpr was equal (difference = 0)
N (negative)The result was negative (s1 < s2 in signed arithmetic)
C (carry)The subtraction produced an unsigned borrow

cmpr sets, jumps read

Always call cmpr immediately before a conditional jump. The flags persist until the next cmpr or arithmetic operation overwrites them. 

cmpr r1, r2   ; compare r1 to r2, set flags
jz   equal   ; jump if r1 == r2
jn   less    ; jump if r1 < r2
jge  big     ; jump if r1 >= r2

Conditional jump reference

InstructionJumps when...Equivalent to
jzZ flag sets1 == s2
jnzZ flag clears1 != s2
jnN flag sets1 < s2 (signed)
jcC flag sets1 < s2 (unsigned borrow)
jgeN flag clears1 >= s2
jleN or Z flag sets1 <= s2

if / else in assembly

High-level if/else translates directly. Compare, branch to the else body if the condition is false, fall through if it's true: 

; if r1 == r2: store 1, else: store 0
    cmpr r1, r2
    jnz  else_branch
; --- then branch ---
    ldi  r3, 1
    st   r3, 0x4000
    jmp  end_if
else_branch:
    ldi  r3, 0
    st   r3, 0x4000
end_if:
    ; continues here

Chapter 8 — Loops

Loops are among the most powerful constructs in any language. In assembly they're just a compare and a conditional jump back to an earlier address.

Count-down loop

The simplest pattern: decrement a counter each iteration, loop while it's not zero. 

ldi  r1, 10       ; loop 10 times
ldi  r0, 0        ; zero register
loop:
    dec  r1
    cmpr r1, r0    ; is r1 == 0?
    jnz  loop      ; if not, go again

Count-up loop with limit

ldi  r1, 0
ldi  r2, 100      ; limit
loop:
    ; --- body here ---
    inc  r1
    cmpr r1, r2
    jnz  loop

Iterating over an array

Use idx as a pointer. Advance it each iteration to walk through memory: 

ldi   idx, 0x4000  ; start of array
ldi   r2, 0x4010  ; end address (16 elements)
ldi   r3, 0       ; sum accumulator
sum_loop:
    ldind r1, idx    ; load current element
    add   r3, r3, r1; add to sum
    inc   idx
    cmpr  idx, r2
    jnz   sum_loop

Nested loops

For nested loops (e.g., iterating over a 2D grid), use separate registers for each counter. Save/restore the outer counter around the inner loop if needed: 

ldi  r1, 0        ; outer counter (rows)
ldi  r5, 128
outer:
    ldi  r2, 0      ; inner counter (cols)
    inner:
        ; --- body: r1=row, r2=col ---
        inc  r2
        cmpr r2, r5
        jnz  inner
    inc  r1
    cmpr r1, r5
    jnz  outer
Infinite loops If your loop condition is never true, the program runs forever. Always double-check that your counter will actually reach the termination value. Off-by-one errors are extremely common here — if your limit is 128, make sure you're comparing after incrementing, not before.

Chapter 9 — The Stack & Functions

The stack is the mechanism for subroutines, temporary variable storage, and preserving register state across function calls. Understanding it is essential for writing any program longer than a few dozen lines.

What is the stack?

The stack is a region of RAM that grows downward from a high address. sp (the stack pointer) always points to the top of the stack. push decrements sp and writes to it. pop reads from sp and increments it. It's LIFO — last in, first out. 

; Before: sp = 0xBFFF
push r1   ; sp = 0xBFFE, RAM[0xBFFE] = r1
push r2   ; sp = 0xBFFD, RAM[0xBFFD] = r2
pop  r3   ; r3 = RAM[0xBFFD] = r2, sp = 0xBFFE
pop  r4   ; r4 = RAM[0xBFFE] = r1, sp = 0xBFFF

Calling functions with call / ret

call addr pushes the address of the next instruction onto the stack, then jumps to addr. ret pops that address and jumps back. This is the call/return mechanism. 

; Calling code:
    ldi  r1, 10      ; argument in r1
    call double       ; call the function
    ; r1 now contains 20
    halt

; Function definition:
double:
    add  r1, r1, r1  ; r1 = r1 * 2
    ret

Preserving registers in functions

Functions often use registers that the caller also uses. To avoid clobbering, push registers you'll modify at the start and pop them at the end: 

my_func:
    push r2          ; save r2 (we're about to overwrite it)
    push r3          ; save r3

    ; ... do work using r2 and r3 ...

    pop  r3          ; restore in reverse order!
    pop  r2
    ret
Pop in reverse order Pushes and pops must be symmetric and in reverse order. Pushing r2 then r3 means you must pop r3 then r2. An unbalanced stack will cause ret to jump to garbage.

Calling convention

YBM-16 has no enforced calling convention, but a consistent one keeps code manageable:

  • Arguments: pass in r1, r2, r3, ... (left to right)
  • Return value: leave in r1
  • Callee-saved: if a function uses r5–r14, it must push/pop them
  • r0: treat as permanently zero (never write to it)

Chapter 10 — The Display

The 128×128 pixel display is the most visible output of any YBM-16 program. Getting comfortable with the pixel instruction and VRAM layout unlocks graphics programming.

The pixel instruction

pixel color_reg, addr_reg writes one pixel of color to the VRAM address in addr_reg. Both operands must be registers — there is no immediate form. 

ldi   r1, 0x000F  ; color index 15 (bright white in default palette)
ldi   idx, 0      ; VRAM address 0 = top-left pixel
pixel r1, idx

VRAM layout

VRAM is a flat array of pixel values starting at address 0x0000 within VRAM space (which is memory-mapped, but the pixel instruction abstracts the mapping). Pixels are stored row by row: 

pixel 0     = (x=0, y=0) top-left
pixel 1     = (x=1, y=0)
...
pixel 127   = (x=127, y=0) end of row 0
pixel 128   = (x=0,   y=1) start of row 1
...
pixel 16383 = (x=127, y=127) bottom-right

Converting (x, y) to a pixel address

addr = y * 128 + x. Since 128 = 2^7, the multiply can be done with a shift: 

; Compute addr for (x=r1, y=r2), result in idx
ldi r3, 7
shl idx, r2, r3   ; idx = y * 128
add idx, idx, r1  ; idx = y * 128 + x

Filling the screen

Fill every pixel in a loop from address 0 to 16383:

ldi   r1, 0x0005  ; color 5
ldi   idx, 0
ldi   r2, 16384
fill:
    pixel r1, idx
    inc   idx
    cmpr  idx, r2
    jnz   fill
halt

Drawing a rectangle

A nested loop over rows and columns:

; Draw a 20x10 filled rectangle at (x=10, y=5)
ldi  r0, 0
ldi  r8, 0x000A  ; color
ldi  r1, 5       ; row = 5
ldi  r9, 15      ; row end = 5+10
row_loop:
    ldi  r2, 10   ; col = 10
    ldi  r10, 30 ; col end = 10+20
    col_loop:
        ldi  r3, 7
        shl  idx, r1, r3
        add  idx, idx, r2
        pixel r8, idx
        inc  r2
        cmpr r2, r10
        jnz  col_loop
    inc  r1
    cmpr r1, r9
    jnz  row_loop
halt

Chapter 11 — Projects

Everything so far has been building toward this. Here are three complete, non-trivial programs that exercise every part of the language. Read through each one carefully — after these, you have seen every important pattern in YBM-16 assembly.

Project 1: Counting Sort

Read 16 values from RAM, count how many are even, store the result. Demonstrates: loops, modulo, conditionals, memory access. 

; Assume 16 values are pre-loaded at 0x4000–0x400F
; Result (count of even numbers) stored to 0x4100

    ldi  r0, 0        ; zero register
    ldi  idx, 0x4000  ; pointer to data
    ldi  r2, 0x4010  ; end address
    ldi  r3, 0        ; even counter
    ldi  r5, 2        ; constant 2

check_loop:
    ldind r1, idx     ; load value
    mod   r4, r1, r5  ; r4 = value % 2
    cmpr  r4, r0      ; is remainder 0?
    jnz   not_even
    inc   r3           ; it's even, count it
not_even:
    inc   idx
    cmpr  idx, r2
    jnz   check_loop

    st    r3, 0x4100  ; store result
    halt

Project 2: Gradient Screen

Fill the screen with a horizontal gradient — each column a different color index from 0 to 15, tiling across all 128 columns. Demonstrates: 2D loops, modulo for wrapping, pixel output. 

    ldi  r0, 0
    ldi  r5, 16       ; palette size
    ldi  r1, 0        ; row
    ldi  r6, 128      ; screen size
    ldi  r7, 7        ; for shl
row_loop:
    ldi  r2, 0        ; col
col_loop:
    mod  r3, r2, r5  ; color = col % 16
    shl  idx, r1, r7
    add  idx, idx, r2
    pixel r3, idx
    inc  r2
    cmpr r2, r6
    jnz  col_loop
    inc  r1
    cmpr r1, r6
    jnz  row_loop
    halt

Project 3: Reusable draw_pixel Function

A callable function that takes (x, y, color) as arguments and draws a pixel. This is the foundation of any larger graphics program. Demonstrates: calling convention, call/ret, register preservation. 

; Convention: r1=x, r2=y, r3=color
; draw_pixel does NOT modify r1, r2, r3

    ldi  r1, 64       ; x = 64
    ldi  r2, 32       ; y = 32
    ldi  r3, 0x000F  ; color = white
    call draw_pixel
    halt

draw_pixel:
    push r4           ; save r4
    ldi  r4, 7
    shl  idx, r2, r4  ; idx = y * 128
    add  idx, idx, r1 ; idx += x
    pixel r3, idx
    pop  r4           ; restore r4
    ret
You're ready You've now seen every major programming pattern YBM-16 supports. From here, combine these building blocks freely. Complex programs are just many small programs wired together with call, loops, and shared memory.

1. Memory & Data

ld dst, addr 0x00

Loads a 16-bit value from a fixed, literal memory address into register dst. The address is embedded directly in the instruction bytes — it does not come from a register. This is a direct memory read: the CPU computes nothing, it simply fetches the word at the given address.

Use ld when you know the address at compile time (e.g., a named variable in RAM). If the address is computed at runtime or lives in a register, use ldind instead.

2. Arithmetic & Logic

add dst, s1, s2 0x06

Adds registers s1 and s2, storing the 16-bit result in dst. Overflow wraps silently (65535 + 1 = 0). All three registers may overlap: add r1, r1, r1 doubles r1.

byte 00x06
byte 1dst<<4 | s1
byte 2s2<<4
byte 30x00
add r3, r1, r2 ; r3 = r1 + r2
add r1, r1, r1 ; r1 = r1 * 2 (double)
sub dst, s1, s2 0x07

Subtracts s2 from s1, storing the result in dst. If s2 > s1, the result wraps (underflow). Use cmpr before subtracting if you need to guard against underflow. 

sub r3, r1, r2 ; r3 = r1 - r2
sub r1, r1, r2 ; r1 -= r2  (in-place)
mul dst, s1, s2 0x08

Multiplies s1 by s2, keeping only the lower 16 bits of the result. For large values the upper bits are silently discarded. If you need to multiply by a power of 2, shl is faster. 

mul r3, r1, r2 ; r3 = r1 * r2 (low 16 bits)
div dst, s1, s2 0x09

Integer division. Divides s1 by s2, storing the quotient (floor) in dst. The remainder is discarded; use mod to retrieve it. Dividing by zero is undefined behaviour — always guard against it if s2 may be zero. 

div r3, r1, r2 ; r3 = floor(r1 / r2)
mod dst, s1, s2 0x0D

Computes s1 modulo s2: the remainder after integer division. Extremely useful for wrapping values (e.g., keeping a coordinate within screen bounds, cycling through palette indices, or testing even/odd with mod 2). 

mod r3, r1, r2  ; r3 = r1 % r2

; Keep x within 0–127:
ldi r5, 128
mod r1, r1, r5  ; r1 = r1 % 128
inc reg 0x0A

Adds 1 to reg in place. Equivalent to add reg, reg, one but more compact. Used constantly in loops and pointer arithmetic. Wraps from 65535 to 0.

byte 00x0A
byte 1reg<<4
byte 20x00
byte 30x00
inc r1   ; r1 = r1 + 1
inc idx  ; advance pointer
dec reg 0x0B

Subtracts 1 from reg in place. The most common loop countdown instruction. Wraps from 0 to 65535. 

dec r1   ; r1 = r1 - 1
neg reg 0x0C

Two's complement negation. Computes 0 - reg and stores the result back. In signed 16-bit arithmetic this flips the sign. Useful for computing differences (a - b = a + neg(b)) or for reversing direction in animation. 

ldi r1, 5
neg r1       ; r1 = 0xFFFB  (-5 in signed 16-bit)
ldi r2, 5
add r3, r2, r1 ; r3 = 5 + (-5) = 0
gand dst, s1, s2 0x0E

Bitwise AND. Each bit of dst is 1 only if the corresponding bits of both s1 and s2 are 1. Primary use: masking — isolating specific bits while zeroing everything else. 

ldi  r2, 0x00FF
gand r3, r1, r2  ; r3 = low byte of r1 only
gor dst, s1, s2 0x0F

Bitwise OR. Each bit is 1 if either (or both) source bits are 1. Use to combine bit fields, set specific bits, or merge two packed values. 

ldi r2, 0x0001
gor r1, r1, r2  ; force bit 0 of r1 to 1
gxor dst, s1, s2 0x10

Bitwise XOR. Each bit is 1 if the source bits differ. Key uses: toggling bits, zeroing a register without ldi (gxor r, r, r), and simple "flip" logic in games. 

gxor r1, r1, r1  ; r1 = 0 (fastest way to zero a register)

; Toggle a boolean flag in bit 0:
ldi  r2, 1
gxor r1, r1, r2
gnot reg 0x11

Bitwise NOT. Inverts all 16 bits of reg in place. Every 0 becomes 1, every 1 becomes 0. Equivalent to XOR with 0xFFFF. 

ldi  r1, 0xFF00
gnot r1           ; r1 = 0x00FF
shl dst, src, count 0x12

Shifts the bits of src left by the number of positions in count, filling the vacated low bits with zeros, and stores the result in dst. Shifting left by N multiplies by 2^N (modulo 65536). Since count must be a register, load the shift amount with ldi first. 

ldi r2, 3
shl r3, r1, r2  ; r3 = r1 * 8  (left shift 3)

; Fast y*128 for pixel address (128 = 2^7):
ldi r4, 7
shl idx, r2, r4  ; idx = y * 128
shr dst, src, count 0x13

Shifts the bits of src right by count positions, filling vacated high bits with zeros (logical right shift). Divides by 2^N (integer, rounding down). Used for extracting the high byte of a packed value, halving numbers, and scaling. 

ldi  r2, 8
shr  r3, r1, r2  ; r3 = high byte of r1 shifted into low position

; Halve a value:
ldi  r2, 1
shr  r1, r1, r2  ; r1 = r1 / 2

3. Control Flow

cmpr r1, r2 0x14

Compares r1 to r2 by performing the subtraction r1 - r2 internally, then setting status flags based on the result. The registers are not modified. No result is stored.

The flags set are:

  • Z — set if r1 == r2 (result is zero)
  • N — set if r1 < r2 in signed arithmetic (result is negative)
  • C — set if r1 < r2 in unsigned arithmetic (borrow occurred)

Always place cmpr directly before the conditional jump that tests its result. Any arithmetic between cmpr and the jump may overwrite the flags.

byte 00x14
byte 1r1<<4 | r2
byte 20x00
byte 30x00
cmpr r1, r2
jz   equal    ; r1 == r2
jn   less     ; r1 < r2
jge  big      ; r1 >= r2
jmp addr 0x15

Unconditional jump. Sets the program counter to addr immediately. Execution continues from there. No flags are tested. Use for infinite loops, skipping over else blocks, and end-of-function cleanup paths.

byte 00x15
byte 10x00
byte 2addr[hi]
byte 3addr[lo]
jmp loop_start
jmp 0x0000      ; jump to program start (restart)
jz addr 0x16

Jump if Zero. Takes the branch if the Z flag is set — meaning the last cmpr found the two operands equal. Falls through to the next instruction otherwise. 

cmpr r1, r0  ; is r1 == 0?
jz   is_zero
jnz addr 0x17

Jump if Not Zero. Takes the branch if the Z flag is clear — the last comparison was not equal. This is the most common loop-continuation check. 

cmpr r1, r2
jnz  not_equal

; Loop pattern:
dec  r1
cmpr r1, r0
jnz  loop      ; keep going while r1 != 0
jn addr 0x18

Jump if Negative. Takes the branch if the N flag is set — meaning cmpr s1, s2 found s1 < s2 in signed arithmetic. Use for signed comparisons and bouncing animation direction. 

cmpr r1, r2
jn   r1_less   ; r1 < r2
jc addr 0x19

Jump if Carry. The carry flag is set when an unsigned subtraction borrows, i.e., when s1 < s2 treating both as unsigned 16-bit values. Use for unsigned less-than comparisons and detecting arithmetic wraparound. 

cmpr r1, r2
jc   unsigned_less  ; r1 < r2 (unsigned)
jge addr 0x1a

Jump if Greater or Equal. Branches when N is clear, meaning s1 >= s2. Complement of jn. Use for "keep going while index is within bounds" or "skip if already at max". 

cmpr r1, r2
jge  at_or_above  ; r1 >= r2
jle addr 0x1b

Jump if Less or Equal. Branches when either N or Z is set, meaning s1 <= s2. 

cmpr r1, r2
jle  at_or_below  ; r1 <= r2

4. Stack & System

push reg 0x1c

Decrements the stack pointer (sp) by 1, then writes the value of reg to the address now pointed to by sp. The register is unchanged.

Use push to save register state before calling a function, or to temporarily store a value when you're out of registers.

byte 00x1C
byte 1reg<<4
byte 20x00
byte 30x00
push r1   ; sp--, RAM[sp] = r1
push r2   ; sp--, RAM[sp] = r2
pop reg 0x1d

Reads the value at the address pointed to by sp into reg, then increments sp. This is the inverse of push. Always pop in the reverse order you pushed. 

pop r2   ; r2 = RAM[sp], sp++  ← pops last thing pushed
pop r1   ; r1 = RAM[sp], sp++
call addr 0x1e

Pushes the address of the instruction immediately following this call onto the stack (the return address), then jumps to addr. The called function uses ret to jump back.

This is the fundamental subroutine mechanism. Functions can call other functions (nested calls) as long as the stack has room — each call uses one stack slot for the return address.

byte 00x1E
byte 10x00
byte 2addr[hi]
byte 3addr[lo]
call my_function    ; push return addr, jump to my_function
; execution resumes here after ret
ret 0x1f

Pops the top value from the stack and jumps to it. This is how a subroutine returns to its caller. The popped value must be the return address placed there by call — if the stack is unbalanced (mismatched pushes/pops inside the function), ret will jump to garbage. 

my_function:
    push r2       ; save r2
    ; ... do work ...
    pop  r2       ; restore r2
    ret            ; jump back to caller
pixel color_reg, pos_reg 0x20

Writes a single pixel to the display. color_reg holds a color index from the 16-color indexed palette (values 0–15; higher bits are ignored). pos_reg holds the linear pixel address: 0 is top-left, 127 is end of the first row, 128 is start of the second row, and 16383 is the bottom-right pixel.

Both operands must be registers. Compute the pixel address from (x, y) with: addr = y * 128 + x, where the multiply can be replaced by shl y, y, 7 since 128 = 2^7.

byte 00x20
byte 1color<<4 | pos
byte 20x00
byte 30x00
ldi   r1, 0x000A  ; color 10
ldi   r2, 64      ; x = 64
ldi   r3, 32      ; y = 32
ldi   r4, 7
shl   idx, r3, r4 ; idx = y * 128
add   idx, idx, r2; idx += x
pixel r1, idx
nop 0x21

No operation. The CPU fetches and decodes this instruction but does nothing. Useful for: timing delays (burn a precise number of cycles), placeholder slots during development, and padding code regions to align addresses. 

nop  ; one cycle of nothing
nop
nop  ; small busy-wait
halt 0x22

Stops the CPU. Execution ends. Register and memory state are preserved for inspection. Every program should end with halt — without it, the CPU will continue fetching whatever bytes come after your code, which is undefined behaviour. 

halt  ; program ends here

Program A: Logic & Conditionals

Checks if a value in memory is even or odd.

    ld   r1, 0x1000   ; Load number from 0x1000
    ldi  r2, 2
    mod  r3, r1, r2   ; r3 = r1 % 2
    cmpr r3, r0       ; Compare remainder to zero (r0 is usually 0)
    jz   is_even
    ldi  r4, 1        ; Store 1 for Odd
    st   r4, 0x1001
    halt
is_even:
    ldi  r4, 2        ; Store 2 for Even
    st   r4, 0x1001
    halt

Program B: Full Screen Scanline

Draws a green horizontal line across the screen.

    ldi   idx, 0       ; Screen buffer start
    ldi   r1, 0x0F00  ; Green color code
    ldi   r2, 640     ; Width of screen

draw_line:
    pixel r1, idx     ; Render
    inc   idx
    cmpr  idx, r2
    jnz   draw_line
    halt

Program C: Function Pointers & Stack

Using call and ret to draw a square pattern.

    ldi   idx, 100     ; Start position
    ldi   r1, 0xFFFF  ; White pixel
    call  draw_dot
    ldi   idx, 110

draw_dot:

    ret               ; Jump back to call site
